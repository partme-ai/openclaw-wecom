# OpenClaw-WeCom 商业增值策略

> 本文档定义 `openclaw-wecom`（`@mocrane/wecom`）作为商业插件的增值定位、三层能力模型、MCP 协同关系与实施路线图。
>
> **定位**：纯加法增值插件——不改原有的双模式 Bot/Agent 消息流逻辑，只在消息流中注入两层核心能力：**知识库 RAG（让 AI 有脑子）** 和 **业务 MCP（让 AI 能干活）**。
>
> **核心卖点一句话**：让你的企业微信 AI 机器人不再只是个聊天玩具——它能索引公司知识、操作企微数据、接入企业私有业务系统。

## 1. 增值定位：纯加法，不侵入

本插件的底层逻辑：**`openclaw-wecom` 本身是一个完整的企微通道插件（双模式 Bot + Agent），商业增值插件不作任何侵入式修改，只在消息流的关键节点上塞入功能：**

```text
消息入站 → [buildInboundBody] → [增值前处理] → Bot/Agent 处理 → [before_prompt_build 钩子] → 回复出站
                                  ↑                                            ↑
                            可在 monitor.ts 的                           注入 RAG 上下文
                            buildInboundBody 之后、                       + MCP 工具定义
                            processInboundMessage 之前
                            插入增值逻辑（纯加法）
```

**什么叫"纯加法"：**

- 不修改 `monitor.ts`、`channel.ts` 等核心文件
- 所有增值功能通过 OpenClaw 的 `before_prompt_build` hook 注入 RAG 上下文和 MCP 工具定义
- 知识库 RAG 和 MCP 扩展通过独立的 `registerKnowledgeHooks()` 和 `api.registerTool()` 注入
- 不破坏原有的 Bot 优先 / Agent 回退逻辑
- 每层增值能力都能独立启用/禁用

**当前已有实现（`feat/knowledge-base-rag` 分支）：**

- **知识库 RAG**：完整骨架（约 1800+ 行 TS），包含文档切片、Embedding、向量存储、混合检索、`before_prompt_build` hook
- **企微 MCP 集成层**（`src/mcp/`）：`tool.ts`（`wecom_mcp` Agent Tool）、`transport.ts`（Streamable HTTP 传输）、`schema.ts`（Schema 清洗）
- **11 个 MCP 对接 Skill**：覆盖文档、通讯录、会议、日程、待办五大品类

**为什么能作为商业插件卖：**

- **通用性强**——知识库、业务对接、群聊整理，几乎每家企业都有这个需求
- **独立可部署**——第一层（知识库 RAG）不依赖任何第三方系统
- **可扩展**——第二层（业务 MCP）通过标准 MCP 协议接入，企业自有系统信创适配即可
- **配置层零代码**——管理员只需在企微里建个文件夹、开个开关

### 纯加法的 5 个合法扩展点

所有增值功能只能通过以下 5 个入口注入，**绝不侵入核心文件**。这对客户意味着：升级主插件时永远不会有冲突，增值功能可独立启用/禁用，稳定性有保障。

| 扩展点 | 技术入口 | 商业用途 | 状态 |
|--------|---------|----------|------|
| **1. 知识库上下文注入** | `before_prompt_build` hook | 在 AI 处理消息前注入 RAG 检索结果和自定义指令 | ✅ 已使用 |
| **2. 工具注册入口** | `api.registerTool()` | 注册 MCP 业务工具，让 AI 能调用企微 API 和企业私有系统 | ✅ 已使用 |
| **3. 消息预处理器** | 增值前处理拦截点（消息解析之后、AI 处理之前） | 消息过滤、审计日志、流量控制、敏感信息检查等 | 🔲 预留 |
| **4. 企微 API 客户端** | 企业微信 API 封装层 | 已封装 AccessToken 管理、媒体上传等基础能力，可供 MCP 层和自定义业务逻辑调用 | ✅ 已有 |
| **5. 技能目录** | 技能（Skills）目录 | 编写 MCP 对接说明书（SKILL.md），指导 AI 正确使用业务工具。当前已有 13 个技能 | ✅ 已有 |

**核心文件保护原则**——以下文件绝不修改核心逻辑：

| 文件 | 原则 | 对客户的意义 |
|------|------|-------------|
| 消息流核心 | ❌ 不修改核心逻辑 | 消息处理不受增值功能影响，Bot 优先 / Agent 回退机制完整 |
| 通道生命周期 | ❌ 不改 | 通道稳定性不受增值功能影响 |
| MCP 传输层 | ❌ 不改核心 | 传输层已完备，新增业务品类无需修改传输代码 |
| MCP 工具定义 | ❌ 不改核心 | `wecom_mcp` 通用工具已支持任意品类扩展，新增品类无需改工具定义 |

**为什么这对客户重要**：纯加法意味着插件的每一层增值能力都可以独立购买、独立安装、独立卸载。今天买知识库，明天加 MCP，后天接 HR 系统——按需搭配，互不干扰，且主插件升级时零冲突。

## 2. 三层能力与 MCP 协同关系

> **核心洞察**：openclaw-wecom 的增值体系是**三层递进**的——从"知道"到"能干"再到"懂业务"。每一层都通过 MCP 协议和能力映射与企微原生能力或企业私有系统协作。

| 层次 | 能力名称 | 核心价值 | MCP 品类映射 |  Skill 覆盖 | 数据源 |
|------|----------|----------|-------------|-------------|--------|
| **L1** | 知识库 RAG | AI 有脑子了 | `doc`（企业文档 MCP） | `wecom-doc`, `wecom-doc-manager` | 企微文档库 → 切片 → Embedding → 向量库 |
| **L2** | 企微 MCP 原生能力 | AI 能干企微的活了 | `contact`, `meeting`, `schedule`, `todo`, `doc` | 共 11 个 Skill | 企微原生 API（通讯录/会议/日程/待办/文档） |
| **L3** | 企业私有业务接入 | AI 懂你的业务了 | 自定义品类（`erp`, `crm`, `hr`, `ticket`, `sql` 等） | 由企业或集成商开发 | 企业自有系统（HR/工单/审批/数据库） |

                    │  RAG 检索    │    │  MCP 工具注入   │

## 3. 第一层：知识库 RAG（"AI 有脑子了"）

不要做成通用 RAG 接入。企微场景中知识库的价值在于回答三类问题：

### 为什么通用

每家公司都有自己的文档、制度、产品说明。不需要对接任何第三方系统，只需要让管理员指定"要索引哪些文档库"。

### 用户视角

```text
管理员：在企微文档库里建一个 "AI 知识库" 文件夹，把制度文档扔进去
Bot：    用户问问题 → 自动搜文档 → 结合上下文回复
管理员：不用了，把文档移出该文件夹即可
```

### 商业包装

- **零配置知识库**：用企微文档库做数据源，不用额外建站、不用写爬虫
- **自动索引**：管理员往指定文件夹扔文档就行，后台自动切片、Embedding、入库
- **无缝集成**：用户无感知，和普通聊天一样问问题，AI 自动判断要不要查知识库

### 具体功能点

#### 数据来源：两条数据通路

知识库 RAG 的数据来源分为两条相互隔离的通路，设计上确保数据不混淆：

| 通路 | 数据源 | 触发方式 | 索引目标 | 使用场景 |
|------|--------|----------|----------|----------|
| **通路 A** | 用户上传文件（对话附件） | 用户发送文件 → Bot/Agent 自动索引 | `accountId:mode` 对话级私有 namespace | 用户与 Bot 共享上下文，不沉淀为全局知识 |
| **通路 B** | 企微文档库（管理员指定） | 定时调度 → 企微文档 API 拉取 | `accountId:enterprise` 企业级全局 namespace | 公司制度、产品文档等需要全员可查的知识 |

> **关键设计约束**：通路 A 和通路 B 写入不同的 namespace，数据不混淆。通路 B 的配置中必须显式指定企微文档库的 `folderId`/`docId`，且需要额外的企微 API 权限（`doc/get_doc_content` 等 MCP Skill）。误将通路 A 的对话文件当作企业知识全量推入，会污染知识库。

| 功能 | 描述 | 实现思路 |
|------|------|----------|
| **企业文档自动索引（通路 B）** | 管理员指定企微文档库 ID（`folderId`/`docId`），通过 `wecom_mcp.call doc get_doc_content` 定时拉取文档内容，切片 + Embedding → 全局知识库（`accountId:enterprise` namespace） | `src/knowledge/indexer/scheduler.ts` 定时任务 + `chunker.ts` 切片 |
| **群聊上下文增强** | @机器人时，检索最近 N 条群聊历史 + 相关文档，将结果注入 system prompt | `src/knowledge/hooks.ts` 的 `before_prompt_build` hook |
| **智能回复溯源** | 回复末尾带引用来源（"参考了《XX制度》第3章"） | 检索结果附带元信息（sourceId + chunkIndex），拼入 system prompt |
| **智能摘要 + 提炼** | 长对话自动提炼要点，存入向量库供后续检索 | 定时任务触发 LLM 摘要 → Embedding → upsert 到向量库 |
| **知识库搜索建议** | 用户输入时，自动建议相关知识库文档标题 | 对知识库文档标题做模糊匹配 + 向量检索 |
| **知识订阅** | 用户可@机器人订阅某类知识更新，文档变更时自动推送通知 | 定时轮询企微文档库变更 → 比对向量库 → 推送群消息 |
| **多租户隔离** | 按 `accountId:mode` 命名空间隔离 + 按数据通路（通路 A 对话级 vs 通路 B 企业级）隔离，不同企业/不同模式/不同来源的知识库互不干扰 | `src/knowledge/hooks.ts` 中的 `namespace` 生成逻辑 + 数据来源路由 |

### 技术架构

```text
┌─────────────────────────────────────────────────┐
│               before_prompt_build hook           │
│  ┌──────────┐    ┌──────────┐    ┌────────────┐  │
│  │ Chunker  │ →  │Embedding │ →  │  Retriever │  │
│  │(文档切片) │    │(向量化)   │    │  (检索)     │  │
│  └──────────┘    └──────────┘    └────────────┘  │
│       ↑                ↓                ↓        │
│  ┌──────────┐    ┌──────────┐    ┌────────────┐  │
│  │ WeCom    │    │ ZVec /   │    │  System     │  │
│  │ 文档库   │    │ SQLite   │    │  Prompt     │  │
│  │ (MCP)    │    │ (向量库)  │    │  (上下文)    │  │
│  └──────────┘    └──────────┘    └────────────┘  │
└─────────────────────────────────────────────────┘
```

**已有实现**（`feat/knowledge-base-rag` 分支，约 1800 行 TS）：

- `src/knowledge/embedding/openai.ts` — Embedding 接口（OpenAI 兼容）
- `src/knowledge/store/zvec.ts` — 纯 JS 向量存储（零依赖，适合开箱即用）
- `src/knowledge/store/sqlite-vec.ts` — SQLite-Vec 后端（适合生产）
- `src/knowledge/indexer/chunker.ts` — 文档切片（按段落/标题/Token）
- `src/knowledge/retriever/hybrid.ts` — 混合检索（向量 + 关键词）
- `src/knowledge/hooks.ts` — `before_prompt_build` hook 注入点（配置合并 + 检索 + 注入 system prompt）

## 4. 第二层：企业 MCP 原生能力（"AI 能干活了"）

> **注意**：本层不是规划，而是**已有现成实现**——`src/mcp/` 层 + skills/ 目录下的 11 个 MCP 对接技能。

openclaw-wecom 已完整对接企微五大品类的 MCP Server，通过 `@mocrane/wecom` 的 `wecom_mcp` Agent Tool 统一调用。

### MCP 集成层架构

| 模块 | 文件 | 职责 |
|------|------|------|
| **Agent Tool** | `src/mcp/tool.ts` | 定义 `wecom_mcp` 工具（list/call 两种操作），参数解析，响应构造 |
| **传输层** | `src/mcp/transport.ts` | Streamable HTTP 协议实现，session 管理，SSE 解析，缓存与错误处理 |
| **Schema 清洗** | `src/mcp/schema.ts` | 内联 `$ref/$defs` 引用，移除 Gemini 不支持的 JSON Schema 关键词 |
| **统一导出** | `src/mcp/index.ts` | 模块导出入口 |

### 已对接的 MCP 品类与 Skill 覆盖

| MCP 品类 | 品类说明 | 对应 Skill 列表 | 能力描述 |
|----------|----------|----------------|----------|
| `doc` | 企业文档 | `wecom-doc`, `wecom-doc-manager` | 文档内容读取、Markdown 编辑、智能表格 Schema 查询 |
| `contact` | 通讯录 | `wecom-contact-lookup` | 通讯录成员查询、按姓名/别名筛选 |
| `meeting` | 会议 | `wecom-meeting-create`, `wecom-meeting-query`, `wecom-meeting-manage` | 创建会议、查询会议列表、管理会议设置 |
| `schedule` | 日程 | `wecom-schedule` | 创建日程、查询日程详情、更新日程、空闲忙闲查询 |
| `todo` | 待办 | `wecom-get-todo-list`, `wecom-get-todo-detail`, `wecom-edit-todo` | 待办列表查询、详情查看、编辑更新 |
| `smartsheet` | 智能表格 | `wecom-smartsheet-schema`, `wecom-smartsheet-data` | 表格 Schema 查询、数据记录读取 |
| `preflight` | 前置检查 | `wecom-preflight` | 消息发送前的能力预检 |

### 在消息流中的注入方式

```typescript
// index.ts 入口 —— 已有实现
api.registerTool(createWeComMcpTool(), { name: "wecom_mcp" });
```

Agent 模型在推理时，通过 `wecom_mcp` 工具的 list/call 操作与企微 MCP Server 交互，完成文档查询、通讯录查找、会议创建等操作。

## 5. 第三层：企业私有业务接入（"AI 懂你的业务了"）

在第一层（知识库 RAG）和第二层（企微 MCP 原生能力）的基础上，第三层解决的是**企业特有的业务系统对接**问题。不是做成通用插件，而是提供**接入框架 + 标准接口**，让企业或集成商可以低成本对接自己的系统。

### 架构模式

```
┌──────────────────────────────────────────────────────────────┐
│                  openclaw-wecom 增值层                        │
│  ┌──────────────┐  ┌──────────────────┐  ┌────────────────┐  │
│  │ L1: 知识库   │  │ L2: 企微 MCP     │  │ L3: 企业私有    │  │
│  │     RAG      │  │    doc/contact/  │  │    HR/工单/审批 │  │
│  │              │  │    meeting/...   │  │    /SQL 查询    │  │
│  └──────────────┘  └──────────────────┘  └────────────────┘  │
│                           │                    │              │
│                           ▼                    ▼              │
│                    ┌──────────────┐  ┌──────────────────┐     │
│                    │  wecom_mcp   │  │  企业私有 MCP     │     │
│                    │ (MCP Server) │  │ (自定义品类)      │     │
│                    └──────────────┘  └──────────────────┘     │
└──────────────────────────────────────────────────────────────┘
```

### 具体接入场景

| 场景 | 描述 | 实现方式 | MCP 工具示例 |
|------|------|----------|-------------|
| **AI 工单系统对接** | 员工@机器人创建工单、查状态、催单 | 注册自定义 `ticket` MCP 品类，对接企业工单系统 API | `ticket.create`, `ticket.getStatus`, `ticket.escalate` |
| **数据查询 MCP（SQL→自然语言）** | 自动写 SQL 查数据库，用自然语言返回结果 | 注册自定义 `sql` MCP 品类，后端接数据库（需 SQL 生成 + 安全审计） | `sql.query`, `sql.explain`, `sql.schema` |
| **审批流集成** | 查审批进度、审批链、待办审批 | 注册自定义 `approval` MCP 品类，对接 OA/审批系统 | `approval.list`, `approval.getDetail`, `approval.chain` |
| **HR 系统对接** | 查假期余额、工资条、组织架构 | 注册自定义 `hr` MCP 品类，对接 HR 系统 API 或数据库 | `hr.leaveBalance`, `hr.payroll`, `hr.orgChart` |
| **CRM 客户查询** | 查客户信息、合同状态、商机进展 | 注册自定义 `crm` MCP 品类，对接 CRM 系统 | `crm.getCustomer`, `crm.listOpportunities` |
| **知识库作为一个 MCP 品类** | 让 AI 能搜索/订阅/推荐知识库内容 | 将知识库本身注册为 `knowledge` MCP 品类 | `knowledge.search`, `knowledge.subscribe`, `knowledge.suggest` |

### 接入原则

1. **MCP 协议统一**：所有私有系统统一注册为 `wecom_mcp` 的自定义品类，复用已有的传输层和 session 管理
2. **安全第一**：私有 MCP 必须做鉴权、审计和 SQL 注入防护
3. **Skill 驱动**：每个私有品类对应一个 Skill 目录（参考 skills/ 下已有 Skill），方便 Agent 选择合适的工具
4. **渐进式接入**：从最简单的"查状态"开始，逐步扩展到"创建/修改"操作

## 6. 三个可落地原型

### 原型 A：HR 助手

```text
用户：@机器人 "我年假还剩几天"
内部流程：
  1. before_prompt_build hook → 检索知识库（员工手册 PDF 中关于年假的规定）
  2. Agent 调用 wecom_mcp call hr getLeaveBalance '{ "userId": "zhangsan" }'
  3. 组合 RAG 上下文 + MCP 返回数据 → LLM 推理 → 回复
Bot 回复："根据《员工手册》第三章，年假按工龄计算。
           您目前剩余年假 3.5 天（总 5 天，已休 1.5 天）。"
```

### 原型 B：群聊知识助手

```text
用户：@机器人 "上次那个微服务降级方案谁有链接？"
内部流程：
  1. 检索群聊历史向量库（最近 N 条已向量化消息）
  2. 检索知识库（文档索引）
  3. 组合上下文 → LLM 推理
Bot 回复："找到了！该方案由 @张三 在上周二（3月15日）分享：
           链接：https://doc.wecom.work/doc/xxx
           方案摘要：基于 Sentinel 的熔断降级，核心策略包括...（以下省略）"
```

### 版本发布策略：两条线独立迭代

知识库 RAG 和能力 MCP 两条线**不捆绑发布**，各自独立迭代。这意味着客户不需要等某一侧的开发完成才能用上另一侧的能力——今天买的知识库能力明天就能用，MCP 能力也随时可用。

```text
时间线 →

  知识库 RAG v0.1      MCP 第一阶段（现有能力）
  (企微文档自动索引)     直接可用，无需等待
      │
      ├── RAG v0.2       MCP 第二阶段（工单系统接入）
      │   (群聊上下文)    可独立交付
      │
      ├── RAG v0.3       
      │   (智能摘要)
      │
      └── RAG v0.4       MCP 第二阶段持续（HR/审批/SQL）
          (知识订阅)
              
  示例发布节奏：
  版本 2026.04 = 知识库 RAG v0.1 上线
  版本 2026.05 = MCP 工单接入 + 知识库 RAG v0.2
  版本 2026.06 = 知识库 RAG v0.3
  版本 2026.07 = MCP HR 接入 + 知识库 RAG v0.4
```

#### 客户视角的版本规划矩阵

| 版本 | 知识库 RAG（客户获得） | 企业 MCP（客户获得） | 一句话价值 |
|------|----------------------|---------------------|-----------|
| **已就绪** | 知识库基础骨架已完备：文档切片、向量存储、混合检索、AI 自动回答 | 全部 5 大品类 MCP 能力可用：文档 / 通讯录 / 会议 / 日程 / 待办 | 开箱即用，今天部署今天见效 |
| **v0.1** | 企微文档自动同步：管理员指个文件夹，AI 自动索引 | 第一阶段能力打标，清晰呈现 | 知识问答 + 工具调用一体化 |
| **v0.2** | 群聊上下文增强：AI 记得群里的历史讨论 | 工单系统对接 | 群聊记忆 + 工单流转 |
| **v0.3** | 长对话智能摘要：AI 自动提炼要点 | HR 系统对接 | 对话压缩 + 人事查询 |
| **v0.4** | 知识库开放接口：搜索 / 订阅 / 推荐 | 审批流 / SQL 查询对接 | 知识能力全面开放 |

**每侧都可独立升级**：客户可以只升级知识库到 v0.2 而不升级 MCP 侧，反之亦然。版本号采用 `vYYYY.MM` 格式，每月至少一个能力点交付，确保客户持续获得价值。

## 8. 商业包装要点

### 定价模型建议

| 层 | 内容 | 建议定价 | 说明 |
|----|------|----------|------|
| 基础 | 双模式 Bot + Agent（开源免费） | 免费 | 社区版，MIT 许可证 |
| 知识库 | RAG 知识库（按文档量/企业数） | 按席位或按文档容量 | 含向量存储、切片、检索 |
| 企微 MCP | 企微原生 MCP 能力（doc/contact/meeting/...） | 捆绑知识库层或单独订阅 | 已有完整实现，只需配置 |
| 企业私有 | 自定义 MCP 品类接入框架 | 按品类数或按调用量 | 需企业自行开发后端 MCP Server |

### 交付模式

- **插件包**：`npm pack` 或私有 registry，通过 `openclaw plugins install` 安装
- **配置驱动**：所有增值功能通过 `openclaw.json` 的 `channels.wecom.knowledge` / `channels.wecom.mcp` 等配置项控制
- **许可证**：商业许可证（非开源），可搭配试用期

### 差异化卖点

| vs 竞品 | 我们的优势 |
|---------|-----------|
| vs 企微原生机器人 | 双模式 + 知识库 RAG + MCP 扩展，一个插件搞定 |
| vs 通用 AI 网关 | 深度绑定企微生态，懂企微的文档、群聊、审批；已有完整 MCP 集成层和 11 个 Skill |
| vs 自建方案 | 开箱即用，不需要开发团队；知识库零配置，MCP 协议标准化 |

### 常见销售场景

1. **HR 场景**："你们的企业微信机器人能查年假吗？" → 原型 A + 私有 MCP
2. **知识管理场景**："我们新员工培训，机器人能回答公司制度问题吗？" → 知识库 RAG
3. **效率提升场景**："我们群里的日报能不能自动汇总？" → 智能摘要 + 表格结构化
4. **业务赋能场景**："机器人能帮我们查工单、催审批吗？" → 私有业务 MCP
